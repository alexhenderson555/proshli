# Telegram Publication of Parsed Vacancies — Design

Reference doc for the Proshli vacancy-publication subsystem. Drafted 2026-05-18 via brainstorming. Builds on the V2 platform spec (`2026-05-17-jobskout-v2-platform-design.md`) and the curated TG sources list (`2026-05-17-tg-channels-curated-list.md`).

## Goal

Distribute filtered parsed vacancies into a Proshli-owned Telegram surface to:
1. Drive traffic to `proshli.ru/{locale}/vacancies/{id}` (a no-cost growth funnel).
2. Establish brand presence in the channel-aggregator landscape where the RU IT audience already lives.
3. Create a passive content layer that grows automatically as the parser ingests more sources.

## Architecture

Hybrid surface, single bot:

- **Channel `@proshli`** — facade. 5 hand-approved top-picks per day, distributed across the workday. Optimized for brand growth and TG search indexing.
- **Forum supergroup `@proshli_jobs`** — 28 topics segmented by role/stack. Filtered firehose: ~30–40% of parsed vacancies pass and auto-publish into the matching topic. Optimized for retention of niche audiences.
- **Bot:** existing Proshli bot (`apps/tgbot/main.py`), promoted to admin in both surfaces. Same bot handles both broadcast posting and DM approval workflow.
- **Publication queue:** new DB table `publication_queue` — FIFO with `topic_id`, `vacancy_id`, `rendered_text`, `status`, `scheduled_for`, `published_message_id`.
- **Worker:** new Celery task `workers.tasks.publisher` (per-15-min batch). Existing Celery beat infrastructure reused.
- **Approval bot extension:** new handlers in `apps/tgbot/main.py` for daily DM with inline ✅/❌ buttons.

## Post format

Extended card, 4-7 lines, ~400-600 chars. Designed so the AI-summary line gives unique value over generic aggregators, and the CTA pulls traffic to proshli.ru.

```
🟢 <b>{title}</b> · {company}
💰 {salary_range_or_dash} · 📍 {location_or_remote}
🛠 {top_3_skills_joined_by_dot}

{ai_summary_one_or_two_sentences}

🔗 <a href="https://proshli.ru/{locale}/vacancies/{id}">Подробнее на Proshli</a>
<i>Источник: {source}</i>
```

Where:
- `salary_range_or_dash` — `300 000 – 450 000 ₽/мес` or `—` if not specified
- `location_or_remote` — `Москва · гибрид` / `Remote (РФ)` / `Remote (worldwide)` etc.
- `top_3_skills_joined_by_dot` — `Python · PostgreSQL · Kafka`
- `ai_summary` — 1-2 sentences (≤200 chars), generated by LLM from full description. Highlights what makes this role interesting (tech, scale, product, team size, etc.). NOT a verbatim copy of the source.
- `source` — `hh.ru` / `Telegram @forpython` / `Career site (Yandex)`. Attribution is non-negotiable — protects against TOS claims and matches RU TG-aggregator convention.

LLM used for `ai_summary`: YandexGPT-Lite for RU mass content (cheap, fast), Claude/GPT for premium summaries if we want to A/B later.

`locale` in the CTA URL — pick from vacancy's primary language (RU vacancies → `/ru/`, EN vacancies → `/en/`).

## Filter rules

Applied at enqueue time. A vacancy is queued only if **all** pass:

1. **Description present:** `len(description) ≥ 100` chars (strips empty/recruiter-stub posts).
2. **Not recruiter agency:** company name does not match recruiter-pattern regex (`/агентств|рекрутинг|recruit|staffing|hr partner|подбор персонала/i`). Whitelist override for known in-house recruiters (Yandex HR, Sber HR — none of these match the regex anyway, but keep override for edge cases).
3. **Has signal:** `salary_min > 0` **OR** description contains role-grade keyword (`middle|senior|lead|principal|старший|ведущий|тимлид`).
4. **Not duplicate:** `canonical_id` not already in `publication_queue` (any status) OR in `published_posts` (final state).

Failures get logged with reason but not retried — the parser will re-emit them only on actual source changes.

`canonical_id`: existing dedup key from vacancy ingestion (title + company + city, case- and whitespace-normalized). One vacancy → one post even if seen via HH + TG + corp site.

## Classification (routing)

Per-vacancy LLM call, returns one of 28 topic IDs. Reuses existing `app/services/llm.py` infrastructure.

Prompt sketch (system + few-shot):
```
You classify IT job vacancies into one of 28 categories.
Categories: [enumerated list with hints from the topics table]
Return ONLY the category number (1-28).
If unsure or out-of-IT, return 28 (Niche).
```

Cost guard: route to YandexGPT-Lite (cheap, ~₽0.002/call at current rates). At 1000 vacancies/day → ~₽2/day → trivial.

Cache classifier output on `vacancy.topic_id` column so re-publish or re-render doesn't re-classify.

Confidence threshold: if model returns a category but confidence is low (no native confidence in YandexGPT — proxy via "does response match expected format"), route to `Niche` (28) rather than guess. Manual reclassification handled via admin UI later (out of scope for this spec).

## Group publication pipeline (Phase 1)

```
[parser emits vacancy]
    ↓
[ingest task] — exists today
    ↓
[publication-prefilter task] — new
    ├── dedup check
    ├── filter rules
    ├── classify → topic_id
    ├── render extended card (LLM)
    └── INSERT publication_queue (status=pending, scheduled_for=now)
    ↓
[publisher Celery beat task, every 15 min] — new
    ├── SELECT FROM publication_queue
    │     WHERE status=pending AND scheduled_for ≤ now
    │     ORDER BY scheduled_for ASC
    │     LIMIT N (rate-limit aware)
    ├── for each row:
    │     ├── TG sendMessage(chat=group, message_thread_id=topic_id, text=rendered)
    │     ├── on success: UPDATE status=published, published_message_id=...
    │     ├── on FloodWait(seconds=X): reschedule scheduled_for = now + X
    │     └── on other error: UPDATE status=failed, attempts+=1, retry up to 3 with backoff
    └── log batch stats
```

Rate-limit math:
- TG: 20 messages/minute per group (across all topics combined), 30 messages/second across all chats (irrelevant at our scale).
- 20/min × 60 min/h × 24 h = 28 800 posts/day theoretical ceiling. We will never approach this.
- 15-min batch with max 100 posts per batch is fine and gives natural throttling.

## Channel approval workflow (Phase 2)

```
[daily 09:00 MSK Celery task]
    ├── compute candidate set: vacancies enqueued in last 24h
    ├── score each: salary_weight (40%) + company_prestige_weight (30%) + freshness (15%) + topic_demand (15%)
    │     - salary_weight: log-normalized salary_max, with cap
    │     - company_prestige: lookup in whitelist (Yandex/VK/Sber/T-Bank/Avito/Wildberries/Ozon/Kaspersky/...) → 1.0; unknown → 0.3
    │     - freshness: hours_since_first_seen, decay
    │     - topic_demand: 1.0 if topic is in top-5 by subscriber count, else 0.5
    ├── pick top-8 by score
    └── bot.sendMessage(chat=admin_dm, inline_keyboard=[✅, ❌] per candidate)
    ↓
[admin presses ✅ on 5]
    ├── on each ✅ click:
    │     INSERT publication_queue (
    │       target='channel',
    │       topic_id=NULL,
    │       scheduled_for = NOW + slot_offset
    │     )
    ├── slot_offset: 10:00, 12:00, 14:00, 16:00, 18:00 MSK — distributes through workday
    └── on ❌: mark candidate dismissed (won't resurface for this admin)
```

Admin DM target: a hardcoded telegram_user_id (Alex's). Future: admin role in DB, multiple approvers.

Rejected candidates can still appear in the forum-group topics — channel rejection ≠ group rejection.

## Database changes

New table `publication_queue`:
```sql
CREATE TABLE publication_queue (
    id BIGSERIAL PRIMARY KEY,
    vacancy_id BIGINT NOT NULL REFERENCES vacancies(id) ON DELETE CASCADE,
    target VARCHAR(16) NOT NULL,         -- 'group' | 'channel'
    topic_id INTEGER,                    -- 1..28 for group; NULL for channel
    rendered_text TEXT NOT NULL,         -- full HTML-formatted post
    status VARCHAR(16) NOT NULL DEFAULT 'pending', -- pending | published | failed | dismissed
    scheduled_for TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    published_message_id BIGINT,         -- TG message_id after publish
    attempts INTEGER NOT NULL DEFAULT 0,
    failure_reason TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    published_at TIMESTAMPTZ,
    UNIQUE (vacancy_id, target)
);

CREATE INDEX idx_pubq_pending ON publication_queue (status, scheduled_for)
    WHERE status = 'pending';
```

New column on `vacancies`:
```sql
ALTER TABLE vacancies ADD COLUMN topic_id INTEGER;
ALTER TABLE vacancies ADD COLUMN classified_at TIMESTAMPTZ;
```

New table `channel_candidates` (Phase 2):
```sql
CREATE TABLE channel_candidates (
    id BIGSERIAL PRIMARY KEY,
    vacancy_id BIGINT NOT NULL REFERENCES vacancies(id) ON DELETE CASCADE,
    score NUMERIC(5,4) NOT NULL,
    presented_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    decision VARCHAR(16),                -- 'approved' | 'dismissed' | NULL
    decided_at TIMESTAMPTZ,
    dm_message_id BIGINT,                -- TG message_id of the approve-prompt
    UNIQUE (vacancy_id)
);
```

New table for company-prestige whitelist (Phase 2, seed manually):
```sql
CREATE TABLE company_prestige (
    company_name_normalized VARCHAR(255) PRIMARY KEY,
    prestige_score NUMERIC(3,2) NOT NULL  -- 0.0 to 1.0
);
```

## Error handling

| Scenario | Behavior |
|----------|----------|
| TG `FloodWaitError` | Reschedule `scheduled_for = now + flood_wait_seconds`, no error log. |
| TG `MessageNotModified` / `MessageEmpty` | Mark `failed`, log, do not retry. |
| TG network timeout | Retry with exponential backoff (60s, 300s, 1800s), max 3 attempts. |
| LLM classifier 5xx / timeout | Skip classification this batch, leave `topic_id=NULL`, will retry on next batch. |
| LLM rendered text exceeds TG 4096 limit | Truncate ai_summary to 100 chars, re-render. Log warning. |
| Bot kicked from group/channel | Mark all pending for that target as `failed` with reason, fire alert via existing structlog pipeline. |
| Vacancy deleted between enqueue and publish | Cascade delete via FK handles it. |

## Phasing

**Phase 1 — Forum group + auto-publish** (~1.5-2 weeks)
- Supergroup setup + 28 topics created via Bot API (one-time script)
- `publication_queue` schema + migration
- `publication-prefilter` task (filter + classify + render + enqueue)
- `publisher` Celery beat task (batch publish to group)
- `vacancies.topic_id` column + LLM classification
- Basic tests: filter logic, classifier prompt, queue state transitions
- **Done when:** new parsed vacancy → appears in correct topic within 15 min

**Phase 2 — Channel + manual approval** (~3-5 days)
- `channel_candidates` schema + migration
- `company_prestige` seed (~100-300 companies)
- Daily 09:00 MSK candidate-selection task
- Bot DM handler for ✅/❌ callbacks
- Channel-publish slot scheduler (5 posts/day, distributed)
- **Done when:** daily approval flow works end-to-end, channel gets 5 posts on workdays

**Phase 3 — Analytics & feedback loop** (later, not in this spec)
- CTR tracking (UTM params on CTA, count clicks via Yandex.Metrika)
- Subscriber growth per topic
- A/B test post formats
- Premium-only filtered channel (whitelist companies, hides salary for free tier, etc.)

## Out of scope

Explicit list — these came up in brainstorming and are deferred:

- **Match-score in posts.** Public posts have no user profile attached. We could show generic "Match for senior IT" tag, but value < complexity. Revisit if data shows it boosts CTR.
- **Backfill of existing parsed vacancies.** Channel/group start empty. Publishing 50k backlog posts at launch would burn TG rate limit and feel like spam to early subscribers. Only new vacancies after launch.
- **Empty-topic notifications.** If a topic has 0 vacancies in 24h, skip silently. No "today nothing here" filler posts.
- **Premium-only stricter filter.** "Whitelist-only" tier (Yandex/VK/Sber-level companies) is interesting as a paid feature but is Phase 4+, not now.
- **Multi-language posts.** Posts are in the vacancy's primary language (RU or EN). No translation layer.
- **Cross-posting to TG channels we don't own.** Out of TOS scope. Posts go only to Proshli surfaces.
- **Reply-to-post engagement.** Forum-group topics will be set to admin-post-only (members can react but not write). No comment moderation needed.
